# 内网穿透：中继架构与成本（研究整合 · 道法自然）

> 本文沉淀手机版内网穿透（"公网穿透"面板）的全部架构研究，作为后续接手者的本源参考。
> 对应实现：`addons/dao-relay/`（Worker 中继）、`addons/rt-flow-app/`（手机版 App）。

---

## 0. 一句话本源

手机在 NAT/蜂窝网络后面，外部（云端 Agent）要驱动它，必须有个"会合点"。本项目提供**两条**会合路线，**默认走路线 A（Worker 中继）**，路线 B 为可选：

| | 路线 A · Worker 中继（默认） | 路线 B · Cloudflare Tunnel（可选） |
|---|---|---|
| 本质 | **自写代码**跑在 Cloudflare Workers + Durable Object 上的应用层消息总线 | Cloudflare **托管产品**（cloudflared ↔ argotunnel 边缘） |
| 谁在中转 | 你账号下一个 Worker，按 `(session,token)` 分片服务所有设备 | 每台设备自起独立隧道，**不经你任何 Worker/账号** |
| 手机侧 | App 内 JS 出站连 `wss://…workers.dev/connect`（无需原生二进制） | 打包的 Go 二进制 `libcloudflared.so` 出站连边缘 |
| 暴露的是什么 | 一条**自定义 RPC 协议** + E2E 信封 | 一个裸 HTTP 服务挂到 `xxx.trycloudflare.com` |
| 连接/协议 | 443 上普通 TLS WebSocket，长得跟正常 HTTPS 一样 | 到 `*.argotunnel.com` 的 QUIC(UDP 7844)/http2，**指纹明显** |
| 国内蜂窝 | ✅ 能过（就是正常 HTTPS/WSS） | ❌ 实测被干扰：cloudflared 拿到域名后退出码 1、外部 530/Error 1033 |
| 计费 | 走你 Cloudflare 账号（当前在免费档内） | 无账号、免费，但国内不稳 |

**结论**：在中国移动网络下，路线 A 是唯一稳定可用的链路，因此**默认且应保持走 Worker**。路线 B 留给能连通 trycloudflare 的场景（海外/部分 WiFi）。

## 1. 为什么没充值还能跑全链路

根因：这套中继用的是 **Workers 免费版 + SQLite 版 Durable Object**，而 Cloudflare 2025 年把 SQLite-backed DO 开放给了免费计划。钥匙在 `wrangler.toml`：

```toml
[[migrations]]
new_sqlite_classes = ["RelayDO"]   # SQLite 版 DO，免费版可用
```

- 官方原文："Workers Free plan: Only Durable Objects with SQLite storage backend are available." 本类 `RelayDO` 正是 SQLite 版 → 命中免费档。若写成 KV 版 DO，免费版根本创建不了。
- `*.workers.dev` 子域本身免费，无需绑定域名/付费。
- **关键：这个中继几乎不写存储**。DO 里只有内存态 `pending`（在途请求 Map）+ 一条 WSS，**没有任何 `state.storage.put`** → SQLite 的行读/行写/存储消耗 ≈ 0。
- 免费档每日额度：Workers 请求 10 万/天、DO SQLite 行读 500 万/天、行写 10 万/天、存储 5GB。日常几台设备、每次操作几条 RPC，离这些线差着十万八千里。

**什么时候会开始要钱？**（规模化）
- 真正先撞的是 **Workers 10 万请求/天** 和 **DO 计算时长**。后者最隐蔽：旧版 WSS 用常驻监听（没上 Hibernation），**每台常连设备 = 一个 DO 全天候计时**，设备一多免费的计算时长会被"常连但没说话"的连接悄悄烧光。
- 一句话：**现在免费 = 规模小 + 不写存储 + 踩中 SQLite 免费档**，不是"永远免费"。

## 2. 扩展性（架构本身很能扛）

- 模型：每个 `(session,token)` 一个独立 Durable Object（`idFromName(relayKey(session,t))`）。**一台手机 = 一个 DO**，彼此隔离、无共享状态、无中心瓶颈，Cloudflare 全球边缘自动分布。
- 含义：10 个用户和 10 万个用户对系统是同一形态——只是多了一堆互不相干的小 DO，**不存在热点/锁竞争**。近乎理想的分片，横向扩展几乎免费。
- 真正的天花板是**钱**，不是架构（见 §1、§4）。

---

## 3. 安全性（够用但有明确缺口）

- **账号数据**：RPC 载荷（邮箱/密码/token）走 **E2E 加密**，Worker 只见密文（零知识）——设计正确。
- **控制面鉴权**：=「知道 `session+token` 即凭证」。token 是每台机随机 128bit、持久化复用（见 §5），熵足够。缺口：
  1. ~~无任何限流~~ → **已加基础限流**（v3.0.0：单实例 10s 窗口 ≤120 次，见 §4）。
  2. `/connect` 谁拿到 token 谁就能当 agent，攻击者可用同一 `session+token` 顶替手机（新连接 `close(1000,"replaced")` 旧连接）→ 可劫持。
  3. CORS `*` + 仅 Bearer，token 一旦泄露即全量接管。
- **现实风险**：token 不要在公开渠道明文粘贴；泄露后到 App「刷新 Token」轮换一次即可使旧 token 失效。
- **加固选项**（按性价比）：
  - 私有模式 `wrangler secret put DAO_TOKEN`（把整条中继锁给固定密钥，企业自托管）；
  - 给 `/relay` 加每请求 HMAC+nonce+时间戳防重放；
  - agent 顶替时要求二次校验。

## 4. 成本优化：WebSocket Hibernation（v3.0.0 已上）

**问题**：旧版 `server.accept()` + `addEventListener` 是常驻监听，DO 在 WSS 连接期间按 wall-clock 时长计费。N 台手机常连 = N 个 DO 全天候计费，成本随在线设备**线性增长**。

**修法**（已实现于 `worker.js`）：迁移到 Hibernation API
- `state.acceptWebSocket(server)` 替代 `server.accept()`；
- `webSocketMessage/Close/Error` 回调替代 `addEventListener`；
- `state.setWebSocketAutoResponse(ping→pong)`：客户端 15s 心跳由**运行时自动应答**，DO 不必从 hibernation 唤醒。

**效果**：连接空闲（无在途请求）时 DO 可被运行时驱逐出内存、**WSS 仍保活**，不再按时长烧 CPU。常连场景成本可降一个数量级。

**协议向后兼容**：`/connect` 出站、`/relay/<session>` POST 驱动、`{type:request|response|ping|pong}` 帧格式**逐字节不变** → 现网手机/驱动方零改动，**仅需重新部署 Worker**。

**限流**：`rateOk()` 单实例（=单设备）10s 滑动窗口 ≤120 次，正常手动操作远不及此，失控/暴力则截断（返回 429）。

**本地验证**：`wrangler dev --local`（workerd）+ Node WS 客户端端到端 8/8 通过：health 版本、no_agent 前置、agent WSS open、ping→自动 pong、公网 POST→agent 回包、错 token→落空实例 no_agent、限流 429。

## 5. 稳定性：token 持久化（P1，v0.15.15 已落地）

**旧陷阱**：token 每次冷启动用 SecureRandom 轮换 → 文档里写死的接入 token 会过期失效，云端 Agent 拿旧 token 驱动会一直 `no_agent`。这是"看起来没变化/连不上"的高频根因。

**修法**（已实现于 `MainActivity.ensureRelayIdentity()`）：
- session + token + e2eKey 均**防卸载持久化、稳定复用**；
- token 仅在**首次生成**或用户手动「刷新 Token」时才更新；
- 当前 `endpoint + token + e2eKey` 自动回写到 `relay-config.json`（引擎 `N.getConn()` 读取），云端可经隧道读「取数指引 MD」拿到最新值。

→ 端点 URL+Token 长期不变，驱动方无需每次重拿。

## 6. 路线 B 实测结论与诚实回退（v0.15.20）

**实测**（Samsung S23 Ultra，移动蜂窝，非破坏性、测完已还原）：
- 开「去中心化隧道」→ cloudflared 起来、拿到 URL `arrival-accompanied-modular-steering.trycloudflare.com`，面板一度显示「已连通」；
- 从云端（墙外）连该 URL → 全程 **HTTP 530 / Cloudflare Error 1033**「无法解析此隧道」；
- 约 30s 后 cloudflared **退出码 1**。
- 本质：cloudflared 到 Cloudflare 边缘的持久连接（http2/QUIC 到 argotunnel）在国内移动网络被干扰/阻断。**这不是代码 bug，是 GFW 对 trycloudflare 的封锁。**
- 关键：**全程手机一直在线没掉**——中继与隧道**并行**（不是替换），隧道失败丝毫不影响中继。

**v0.15.20 诚实回退实现**（`RelayService.onTunnelExit()`）：
- cloudflared 异常退出且隧道仍开启 → 自动重试 N 次（退避 2s/4s/6s）；
- 超过 `TUNNEL_RETRY_MAX=3` 次 → 设 `fallback=true`，面板明示「去中心化隧道不可用 · 已回退中继(Worker)，手机仍在线可远程接入」；
- 面板新增「当前生效链路」指示：隧道连通→`🌐 去中心化隧道（直连本设备）`；否则→`☘ 中继 Worker（默认）· 在线`。

## 7. 优先级建议（上量前）

1. **DO Hibernation**（省钱+稳）——✅ v3.0.0 已做。
2. **token 稳定化/自动回写**（治"连不上"）——✅ v0.15.15 + v0.15.20 已做。
3. **限流**——✅ v3.0.0 基础版已做；如需更强可叠加私有模式/HMAC。

## 8. 关键事实速查

- 公网入口（默认中继）：`https://dao-relay-do.zhouyoukang.workers.dev/relay/<session>`
- 协议：出站 `WSS /connect?session=<id>&token=<t>`；入站 `POST /relay/<session>`（Bearer token）
- 帧：`{type:"request",id,path,method,body}` / `{type:"response",id,status,body}` / `{type:"ping"}` / `{type:"pong"}`
- Worker 源码：`addons/dao-relay/worker.js`（默认处理器 + `RelayDO`）、`keys.js`（`VERSION`/`relayKey`/`sharedTokenOk`）
- 重新部署：`cd addons/dao-relay && npx wrangler@3 deploy`（或 `npm run deploy`）
- 本地自测：`npx wrangler@3 dev --local`，再跑 Node WS 端到端脚本
